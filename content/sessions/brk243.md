## 概要

Microsoft Foundry 上でのエージェントハーネスの設計パターンとデプロイ方法を解説する Expert レベルセッション。Foundry の 3 層アーキテクチャ（Intelligence → Runtime → Human-Agent Collaboration）を軸に、Claw スタイルの長時間稼働エージェント（Hermes）のクラウドデプロイ、Microsoft Agent Framework による C# でのカスタムハーネス構築、Foundry から Teams / M365 Copilot への公開を 3 つのデモで実演した。

前日の Tina & Jeff による Foundry 概要セッションのフォローアップとして、より深い技術的実装にフォーカス。

## キーポイント

1. **エージェントハーネスの定義**: エージェントを「モーター」とすると、ハーネスは「車」。コンテキスト管理、ツール/スキル、オーケストレーション、メモリ、ライフサイクルフック、ヒューマンインザループの 6 コンポーネント
2. **Hermes を Foundry Hosted Agents にデプロイ**: Claw スタイルエージェントをクラウドでステートレス運用。セッション ID でファイルシステムを分離、Routines 機能で夜間メンテナンスを自動化
3. **Microsoft Agent Framework が GA（v1.0）**: Python と C# で提供。Agent Loop + Workflows + Harnesses の 3 層構成。Copilot SDK / Claude Code SDK 統合、CodeAct パターンも対応
4. **AG-UI 統合**: オープンスタンダードの AG-UI プロトコル + Copilot Kit で数行のコードで UI 構築
5. **Autopilot エージェント**: 新しいエージェントカテゴリ。ユーザーアカウントを持ち、メール送信・Teams メッセージ・ドキュメント作成を自身の名前で実行可能
6. **Foundry → Teams / M365 Copilot のワンクリック公開**: Entra ID + テナント管理者ガバナンス付き。ストリーミング・引用・推奨アクション・添付ファイルをネイティブサポート。Bot Framework ボイラープレート不要

## 詳細

### エージェントハーネスとは（Shawn Henry）

セッション冒頭でエージェントハーネスの概念を定義:

- **Agent Loop（コアエンジン）**: コンテキスト + ツール → LLM 呼び出し → ツール実行 → 繰り返し。ゴール到達で終了
- **コンテキスト管理**: プロンプト・メッセージ・ツール・スキルの蓄積を制御。コンテキストコンパクション機能で管理
- **ツールとスキル**: MCP / OpenAPI のツール + テキストベースのスキル。特にファイルシステム・コード実行・シェル実行など「コンピュータを与えるツール」が強力
- **エージェントオーケストレーション**: ウェブブラウジング・コード実行・ドキュメント合成など専門エージェントの管理
- **メモリとセッション永続化**: セッション跨ぎの学習、ランタイムのステートレス化を実現
- **ライフサイクルフック**: LLM 呼び出し前後、ツール呼び出し前後にポリシー適用
- **ヒューマンインザループ**: 高リスク操作時にユーザー確認を求めるチャネル提供

Claw スタイルの特徴: ハーネス + メモリ + サンドボックス/VM + 長時間自律実行 + コミュニケーションプラットフォーム統合。OpenClaw が普及させたパターンであり、Hermes はその独自実装。

### デモ 1: Hermes を Foundry にデプロイ（Glenn Condron）

Hermes（Claw スタイルエージェント）を Foundry Hosted Agents 上で運用するデモ:

- **構成**: ローカル TUI（ターミナル UI）→ Foundry Hosted Agents 上の Hermes バックエンド。Azure Default Credential で認証
- **セッション ID によるファイルシステム分離**: セッション ID を変えると完全に新しい Hermes インスタンス（新しい VM、新しいファイルシステム）が起動
- **Work IQ MCP 連携**: Toolbox Connection 経由で SharePoint ドキュメントにアクセス（デモ中はネットワーク問題で一時的に失敗したが構成を説明）
- **Routines 機能（Preview）**: エージェントが自分自身のメンテナンスルーチンを作成。夜間にスキル整理、古いスキル削除、バックアップを自動実行
- **ステートの設計判断**: 30 日間非活動で sandbox が削除される Foundry の仕様を前提に、夜間 Blob バックアップで永続化。「ペットを飼うか、家畜として管理するか」の設計トレードオフ
- **15 分アイドルでシャットダウン**: コスト最適化のためエージェントはアイドル時に停止、Routine でメンテナンス時のみ再起動

Glenn のキーメッセージ: 「エージェントの状態はユニークになるよう設計する（スキル・コンテンツの蓄積）。そのユニークさがリカバリの課題を生むので、状態管理の戦略を事前に決めよ」

### デモ 2: Microsoft Agent Framework でカスタムハーネス構築（Shawn Henry）

C# による Microsoft Agent Framework の agent harness 機能のデモ:

#### Agent Framework の機能ステータス（Slide 26）

| # | コンポーネント | ステータス |
|---|--------------|----------|
| 1 | Agent Loop | Agent Harness Loop **Public Preview** |
| 2 | Context Management | **v1.0** |
| 3 | Skills and Tools | Skills **v1.0** |
| 4 | Sub-agents | Auto sub-agents **Preview** |
| 5 | Session persistence (memory) | Workflows **v1.0** |
| 6 | System prompt assembly | File-based memory pipeline |
| 7 | Lifecycle hooks | Agent middleware |
| 8 | Permissions and safety | Tools permission **Preview** |
| 9 | HIL and planning | **Preview** with Harness |

新機能（Slide 27）: Copilot SDK Integration（**Release**）、Claude Code SDK Integration（**Preview**）、Orchestrations Workflows Patterns（**Release**）、CodeAct（**Preview**）

#### Agent Framework の 3 層構成

1. **Agent Loop**: Foundry モデル + Foundry ツール + 外部プロバイダ（OpenAI、Anthropic、Gemini、Bedrock、Ollama）。デプロイ先も Foundry / Azure Functions / Container Apps / AWS を選択可能。コネクタで Foundry Prompt Agent / Copilot Studio / Claude Code / GitHub Copilot CLI / A2A プロトコルに接続
2. **Workflows**: Sequential、Handoff、Author-Critic、Magentic（MS Research 共同開発のプランニング型）。カスタムワークフローは YAML 宣言的定義も可能
3. **Harnesses**: 共通ツール（ファイルシステム・コード実行・シェル）、コンテキスト構築、プランニング、ミドルウェア（コンパクション、ツール選択、権限管理）

#### コードパターン

- `ChatClient` → `.AsAgentHarness()` で最小ハーネスを即座に構築（2 行追加）
- カスタマイズ例: `Name`, `Description`, ツール追加、`ReasoningEffort`, `FileAccessStore`, メモリ設定, テレメトリ出力先, バックグラウンドエージェント
- **Harness Console**: TUI ベースのプロトタイピング環境。plan/execute モード、TODO リスト、セッションエクスポート
- **AG-UI 統合**: `app.EnableAgUI()` の 1 行で AG-UI エンドポイントを有効化。Copilot Kit のライブラリで即座にチャート・コントロール付き UI
- **Foundry デプロイ**: `AgentHostBuilder` + Responses エンドポイントで直接デプロイ。Foundry Toolkit Agent Inspector でデバッグ可能

### デモ 3: Teams / M365 Copilot への Autopilot エージェント公開（Amanda Foster）

Foundry から Teams / M365 Copilot への公開フローと Autopilot エージェントの新概念:

#### Foundry の 3 種類のエージェント

| タイプ | 特徴 | ID の使い方 |
|--------|------|------------|
| Assistive | ユーザーの拡張として動作（メール送信、Teams メッセージ代筆） | ユーザーの代理 |
| Autonomous | バックグラウンド実行、Azure リソースへのアクセス権限付与可能 | 独自 ID だがユーザーアクション不可 |
| **Autopilot**（新） | 独自ユーザーアカウント。メール・Teams メッセージ・ドキュメント作成を自身の名前で実行 | 完全な独自ユーザーアカウント |

#### ワンクリック公開フロー

1. Foundry ポータルで「Publish to Teams and M365 Copilot」をクリック
2. エージェント名・説明・アクション一覧を入力
3. 「自分だけ」（即座に利用可能）or 「組織に提出」（Admin Center で承認後に全社展開）を選択
4. Teams と M365 Copilot の両方に自動的に登場（別途 UI 構築不要）

#### Work Stream Manager サンプル

グループチャット向け Autopilot エージェントのデモ:
- オンボーディングフロー: エージェント作成者に「誰と話していいか」を確認
- アクセス制御: 許可されていないユーザーには応答しない
- グループチャットでの振る舞い: @メンション時のみ応答、全メッセージには反応しない
- オープンアイテム追跡: チャット内のタスクを自動追跡
